## URL归一化工具 - 参数位置规律研究

通过web搜索验证了URL参数位置的经验规律:
- **核心发现**: 追踪参数(utm_*、fbclid等)通常由营销/广告平台在用户点击分享链接时"追加"到URL末尾
- **技术原因**: 追踪参数是作为"非核心的补充信息"附加在路径之后,确保网页内容本身不变
- **三层参数结构**:
  - 前面: 核心业务参数(id、v、q等) - 网站后端生成链接时就有
  - 中间: 功能性参数(lang、sort、page等) - 用户操作或网站功能添加
  - 后面: 追踪参数(utm_*、fbclid等) - 营销/广告平台在分享时追加
- **实际例证**: 
  - 原始链接: `https://example.com/product?id=123&color=red`
  - 从Facebook点击: `https://example.com/product?id=123&color=red&fbclid=xyz&utm_source=facebook`
  - 每次分享都在末尾追加新参数
- **策略验证**: 这证实了穷举模式"从后往前删除"策略的合理性,符合80/20法则(80%的无用参数集中在URL末尾)

## URL归一化工具 - 新增Fragment参数支持(v3.2)

用户发现工具无法处理`#`后的Fragment参数(如`https://www.iso.org/obp/ui/en/#!iso:std:57180:en`),要求实现统一处理所有参数的功能。

**核心思路**: 参数扁平化 + 统一穷举,不区分参数来源(查询参数或Fragment参数)

**技术实现**:

1. **统一参数提取** - 新增`extractAllParams(url)`函数:
   - 提取查询参数(URLSearchParams)
   - 解析Fragment内容(#之后):
     - 标准锚点: `#section`
     - Hashbang路由: `#!route`
     - Fragment参数: `#key1=val1&key2=val2`
   - 返回统一的参数数组,每个参数包含type、key、value、display

2. **统一URL重建** - 新增`buildURLFromParams(baseUrl, paramsToKeep)`函数:
   - 根据保留的参数重建完整URL
   - 分别处理查询参数和Fragment参数
   - 保持原始顺序和格式

3. **可视化区分**:
   - 参数分析面板: 🔵查询参数 / 🟣Fragment参数
   - 穷举模式: 统一按原始位置顺序从后往前删除
   - 智能分析模式: Fragment参数同样参与评分和过滤

4. **技术要点**:
   - URL标准结构: `protocol://domain/path?query#fragment`(顺序不可颠倒)

   - 同时支持`?`和`#`的混合URL

- 智能分析: 识别并评分Fragment参数

## URL 智能归一化工具 - Fragment 参数行为发现

用户发现一个有趣的现象（以 `https://www.iso.org/obp/ui/en/#iso:std:iso:tr:14121:-2:ed-2:v1:en` 为例）：
- **场景1**：先打开完整URL，页面加载完成后手动删除Fragment参数 → 页面**不变**
- **场景2**：直接打开删除Fragment后的URL → 页面**变化**（无法显示正确内容）

**原因分析**：
1. Fragment参数（`#`后的内容）完全不会发送到服务器，仅由浏览器端JavaScript处理
2. SPA（单页应用）的前端路由机制：JavaScript在页面`load`事件时读取Fragment并渲染内容
3. `load`事件只在页面首次加载时触发一次，后续手动修改地址栏不会触发重新加载
4. 因此：
   - 场景1：页面已渲染完成，修改地址栏不会触发重新加载 → 内容保持不变
   - 场景2：页面加载时没有Fragment参数，JavaScript无法找到要渲染的内容 → 显示默认页面或空白

**工具价值验证**：
- 工具的iframe预览机制正好模拟"删除后再打开"的场景（场景2）
- 能准确测试出删除参数后重新访问时页面是否还能正常显示
- 这才是实际用户分享链接时的真实场景
